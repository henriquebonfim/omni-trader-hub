How the Pipeline
Actually Works

A complete walkthrough of every component, data flow, CLI integration, and decision point — from open issue to shipped release.

The Full Pipeline

8 stages. Issues in, shipped releases out. Every stage is automated, validated, and traceable.

01
Strategic Context Scan

Reads TODO.md + README, lists open PRs, measures velocity from recent merges. Prevents creating duplicate work before anything starts.

gh pr list gh issue list
02
Issue Triage & Scoring

Fetches all open issues as JSON, classifies each as BUG/FEATURE/REFACTOR/etc., scores each on 4 axes (Impact, Effort, Dependency Unblock, Risk). Produces a ranked matrix. Posts a structured status comment back to every issue.

score_issues.py post_issue_comments.py issue-matrix.json
03
Branch Creation & Implementation

Creates a feature branch (never touches main/master/production). Runs /handle-code for each confirmed issue in priority order. Every commit includes Closes #N. CHANGELOG.md is updated automatically.

software-engineer-worker git checkout -b
04
Jules Handoff (async tasks)

Complex or parallel tasks are handed to Jules with an enriched description: problem context, acceptance criteria, constraints, and definition of done. Jules gets the top-ranked task from TASKS.md — not a vague one-liner.

jules new --repo jules remote list jules teleport
05
Batch PR Creation

After all confirmed tasks are implemented, a single PR is created referencing every closed issue. The PR body is auto-generated from the issue matrix — every Closes #N reference is included. TASKS.md is also generated for the Jules handoff record.

gh pr create TASKS.md
06
Code Review

Two modes: /handle-pr-review (read-only analysis — checks out PR, runs build+lint+tests, posts a structured batched review via GitHub API). /handle-pr-code (implements every actionable review comment, posts status replies, creates follow-up issues for out-of-scope items). Both use the same runtime-summary.json baseline to detect regressions.

post_review.py post_comment_replies.py gh api pulls/reviews
07
Merge & Cleanup

/handle-close-pr validates CI, checks review approval, merges (respecting repo merge strategy), deletes the remote + local branch, prunes remote refs, and cleans all tmp/ artifacts. The working tree is confirmed clean before finishing.

gh pr merge --delete-branch git remote prune
08
Release

/handle-release classifies all commits since the last tag by conventional prefix, recommends a SemVer bump, generates a CHANGELOG entry and a polished GitHub release body, bumps the version file, commits, tags, pushes, and calls gh release create. Dry-run mode available for preview.

generate_release_notes.py gh release create git tag -a

The pipeline is a loop — after Step 8, return to Step 3 (priority scoring) and assign the next task. Velocity data from completed cycles informs future scheduling.

/start-workflow

The master orchestrator. Runs everything in sequence. Loops until all issues are resolved and a release is shipped.

1
Context Scan

Reads TODO.md, README.md head, lists open PRs with branches, lists last 10 merged PRs for velocity. Gets open issue count grouped by label. Prevents creating duplicate work.

2
Issue Processing → /handle-issues

Full triage cycle. Outputs TASKS.md with scored, ranked confirmed tasks.

3
Priority Review

Reviews TASKS.md. Cross-checks against active PRs. Checks if Jules already has sessions running. Confirms top task with user (or auto-proceeds).

4
Jules Assignment

Builds enriched task description including: problem context, acceptance criteria (5+ checkboxes), technical constraints, definition of done. Calls jules new --repo owner/repo "description". For parallel workstreams: --parallel 2.

5
Monitor Jules Session

Polls jules remote list --session. When Jules creates a PR, either jules remote pull --session N --apply to apply patch locally, or jules teleport N to clone+checkout+apply in one command.

6
Review → /handle-pr-review or /handle-pr-code

Read-only review first. If changes needed: gh pr comment N --body "@jules <specific correction>" to re-trigger Jules, OR /handle-pr-code to implement directly.

7
Merge → /handle-close-pr

Validates CI, confirms approval, merges with branch deletion, prunes remote refs.

8
Release → /handle-release (when milestone complete)

Counts unreleased commits since last tag. If significant, triggers release pipeline.

↺
Loop

Return to Step 3. Pick the next ranked task from TASKS.md. Repeat.

Priority Scoring Formula

Each confirmed issue is scored across 4 axes, max 50 points:

score_issues.py — scoring logic 
# Axis 1: User Impact (0–10, weight ×2)
impact = 10  # crash, data loss, security, auth, payment
impact = 7   # error, fail, cannot, wrong
impact = 5   # default
impact = 2   # docs

# Axis 2: Effort inverse (0–10, quick wins score higher)
effort = 9   # typo, rename, small change
effort = 5   # default
effort = 2   # refactor, rewrite, migrate, overhaul

# Axis 3: Dependency Unblock (0–10, weight ×1.5)
unblock = 7  # "blocks" in body
unblock = 3  # references other issues
unblock = 0  # standalone

# Axis 4: Risk if Delayed (0–10)
risk = 10    # security, CVE, XSS, injection
risk = 6     # BUG classification
risk = 1     # DOC only

# Final score (max = 50)
score = (impact × 2) + effort + (unblock × 1.5) + risk

/handle-code

The implementation engine. Every code change goes through here, whether triggered by an issue, a PR comment, or direct user request.

0
Stack Auto-Detection

Detects: runtime (Node/Python/Go/Rust), package manager (npm/pnpm/yarn/pip/cargo), test framework (jest/vitest/pytest/cargo test), linter (eslint/ruff/biome), type checker (tsc/mypy/pyright), CI workflows. Stores as internal commands: test_cmd, lint_cmd, typecheck_cmd, build_cmd.

1
Thinking Protocol

Restates the request, traces the call graph (reads actual source), classifies as feat/fix/refactor/perf/test/docs/chore, identifies risks (breaking changes, concurrency, security boundaries, migrations). No coding until complete.

2–3
Discovery → Design

Reads source files, traces call graphs, checks git log on affected files, finds existing patterns to follow. Produces a design table: files to modify, new function signatures, data structures, test strategy, backward compatibility strategy.

4
Veto Checkpoint

Hard stops: arch boundary violation? premature abstraction? scope creep? security boundary crossed? public API broken without versioning? Any veto → redesign before proceeding.

5
Implementation

No any, no implicit types, no unchecked nulls. Functions <50 lines. All external input validated at boundary. All async wrapped with error handling. Commit format: type(module): description\n\nRefs: #issue

6
Documentation Update (always)

CHANGELOG.md updated under [Unreleased] — always. README if user-facing behavior changed. Docstrings/JSDoc for modified public functions. .env.example if new env vars added. Creates CHANGELOG.md if it doesn't exist.

7
Self-Correction Loop

Runs: lint → typecheck → test suite → build. Classifies failures: TypeMismatch → fix annotations. TestFailure(regression) → fix root cause, never fix tests to hide bugs. ArchViolation → pause, redesign. Max 3 attempts on same failure before marking BLOCKED.

8
Final Validation

git diff --stat HEAD to confirm minimal scoped diff. git status to confirm no unrelated changes or temp files. Reports: type, files changed, tests added/updated, docs updated.

/handle-pr-review

Read-only code review. Validates the full runtime, analyzes every changed file, and submits a single batched GitHub review. Never modifies any code.

0
Checkout + Validate Branch

gh pr checkout N → confirms current branch matches PR head ref. Runs full build+lint+typecheck+tests. Stores PASS/FAIL/N/A in runtime-summary.json. If failures: adds HIGH severity findings for each failing module — does NOT fix.

1
Discovery

Fetches: PR metadata, changed file list (gh pr diff N --name-only), full diff (gh pr diff N), existing review threads (gh api pulls/N/comments). Avoids duplicating already-resolved comments.

2
File-by-File Analysis

Analyzes only changed lines. Detects: null/undefined access, unchecked external input, unsafe async, missing error handling, security exposure (injection, IDOR, missing auth), N+1 queries, architectural violations, missing test coverage. Each finding → {path, line, severity: HIGH/MEDIUM/LOW, category: BUG/SAFETY/SECURITY/PERF/ARCH/TEST, message}.

3
Risk Classification

Overall PR gets: SAFE (no findings, all green) / LOW RISK (only LOW findings) / MEDIUM RISK (MEDIUM findings or test failures) / HIGH RISK (any HIGH finding, security issue, or runtime failure).

4
Batched Review Submission

post_review.py N builds a review payload with inline comments + review body. Event: APPROVE (no findings, different author) / REQUEST_CHANGES (findings present) / COMMENT (own PR). Submits via gh api repos/.../pulls/N/reviews --method POST --input payload.json.

5
Verify + Cleanup

git status must be clean. git log must be unchanged from checkout. Wipes all tmp/ files.

/handle-close-pr

Validates, merges, and fully cleans up a PR. Never merges failing or unapproved PRs.

1
Pre-merge Validation

Checks: PR state (must be OPEN), mergeable status, mergeStateStatus (CLEAN required), all CI checks passing via gh pr checks N, review decision. Detects repo merge strategy (merge/squash/rebase) from gh repo view --json mergeCommitAllowed....

2
Merge with Branch Deletion

gh pr merge N --merge --delete-branch. The --delete-branch flag removes the remote branch atomically with the merge. No separate cleanup step needed for remote.

3
Local Cleanup

git checkout main && git pull origin main. Then git remote prune origin to remove stale tracking refs. Confirms local feature branch is gone.

4
Jules Session Note

Lists active sessions via jules remote list --session. Jules sessions expire automatically — no explicit deletion needed. Documents which session ID was used for traceability.

# 2. Generate notes

  --version 1.4.0

# 3. Bump version file (example: package.json)

# 4. Insert changelog entry + commit

# 5. Tag + push

git push origin HEAD
git push origin v1.4.0

# 6. Create GitHub release

  --title "v1.4.0" \
  --notes-file .agent/skills/release-manager/tmp/release-body.md \
  --latest

gh CLI Integration

Every GitHub operation uses the gh CLI. Here's every command used, where, and why — no web UI simulation.

gh
Issues
  • gh issue list --state open --limit 100 --json number,title,body,labels,assignees,milestone,createdAt,updatedAt — fetch all open issues as machine-readable JSON
  • gh issue view N --json number,title,body,comments,labels — read full issue for engineering validation
  • gh issue comment N --body "..." — post structured status reply
  • gh issue close N --reason completed|not-planned --comment "..." — close resolved/invalid issues
gh
Pull Requests
  • gh pr list --state open|merged --json number,title,body,headRefName,mergedAt — detect active PRs and recent merges
  • gh pr checkout N — check out PR branch for review or implementation
  • gh pr view N --json number,title,headRefName,mergeable,statusCheckRollup — pre-merge validation
  • gh pr diff N --name-only — changed files for scoped analysis
  • gh pr checks N --watch — wait for CI to finish
  • gh pr create --title --body --label — create batch PR with Closes refs
  • gh pr merge N --merge --delete-branch — atomic merge + branch cleanup
  • gh pr comment N --body "@jules ..." — re-trigger Jules on a PR
gh
API (direct)
  • gh api repos/{owner}/{repo}/pulls/N/comments — fetch ALL inline review comments with IDs
  • gh api repos/{owner}/{repo}/pulls/comments/{id}/replies -f body="..." — reply to specific inline comment thread
  • gh api repos/{owner}/{repo}/pulls/N/reviews --method POST --input payload.json — submit batched review with all inline comments in one API call
  • gh api repos/{owner}/{repo}/pulls/N/reviews — fetch existing review threads to avoid duplicates
gh
Releases & Repo

  • gh release view vX.Y.Z — verify release was created correctly
  • gh repo view --json nameWithOwner,mergeCommitAllowed,squashMergeAllowed,rebaseMergeAllowed — detect merge strategy

  • gh issue create --title --body --label follow-up — create tracking issues for out-of-scope PR comments

Jules CLI Integration

Jules handles async, complex, and parallelizable engineering tasks. The pipeline assigns Jules the top-priority task with a rich description — not a vague one-liner.

J
Assigning Tasks
  • jules new --repo owner/repo "enriched task description" — assign a single task. Jules gets: problem context, acceptance criteria, technical constraints, definition of done.
  • jules new --repo owner/repo --parallel 2 "task" — spin up 2 parallel sessions for the same task when exploring solutions
J
Monitoring Sessions
  • jules remote list --session — list all active sessions and their status
  • jules remote pull --session N — view session result without applying
  • jules remote pull --session N --apply — apply Jules' patch to local repo for local review
J
Teleport
  • jules teleport SESSION_ID — if not in a repo: clones the repo, checks out Jules' starting branch, applies the patch. If already in the repo: applies the patch directly. One command to get Jules' work onto your local machine.
J
Feedback Loop
  • When Jules creates a PR, check it with /handle-pr-review N
  • If changes are needed: gh pr comment N --body "@jules <specific correction>" re-triggers Jules on the same PR
  • If direct fix is faster: /handle-pr-code N implements the changes directly

The quality of Jules' output is directly proportional to the quality of the task description. The pipeline builds descriptions with: problem context from the issue body, 5+ acceptance criteria checkboxes, technical constraints (follow existing patterns, conventional commits), and a definition of done checklist. Jules gets a spec, not a sentence.

Rules & Standards

4 rule files in .agent/rules/. These are model_decision triggers — they activate automatically when the relevant workflow runs.

📐
software-engineering-standards
trigger: any code modification

Architecture discipline (layer separation, no circular deps), strict typing (no any/implicit/unchecked nulls), code quality (functions <50 lines, no duplication), testing requirements (every public fn + every bug fix), security (validate input, no hardcoded secrets), CHANGELOG discipline. Explains the WHY behind each rule — not just MUST/NEVER commands.

🎯
issue-task-standards
trigger: processing issues

Branch safety enforcement (never commit to main/master/production), mandatory PR creation with Closes #N references, gh CLI-only GitHub operations, temp artifact control (all files in skills/tmp/, gitignored, wiped after), per-issue status patterns (exact format for each status type).

💬
pr-code-standards
trigger: implementing PR feedback

Branch discipline (checkout → validate → never force push), runtime validation before any change (baseline must be stored), commit standards (conventional commits, one per logical fix, PR comment URL in body), comment reply pattern (exactly one reply per thread, never silent close), scope control (create new issue instead of expanding scope).

🔍
pr-review-standards
trigger: read-only code review

Absolute no-modification rule (no code edits, no commits, no branch creation), runtime validation mandatory, review comment pattern (one comment per issue, structured format), risk classification (SAFE/LOW/MEDIUM/HIGH RISK), gh CLI only (gh api for batched review submission).

Add the 7 lines from _gitignore-additions.txt to your .gitignore once. Every skill's tmp/ directory will be protected from accidental commits for all future runs.
