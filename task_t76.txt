T76 - Improve runtime reporting surface

## Task Description
Enhance the runtime reporting capabilities of OmniTrader by expanding the API surface for operational metrics, performance data, and system health indicators. The current /api/status endpoint provides basic bot state, but we need richer reporting for monitoring, debugging, and analytics.

## Current State Analysis
- /api/status: Basic bot running state, uptime, symbol, paper mode, circuit breaker, WS clients
- Limited to core operational flags
- No performance metrics, trade statistics, or detailed system health

## Required Improvements
1. **New /api/metrics Endpoint**: Comprehensive runtime metrics including:
   - Trade performance (win rate, P&L, volume)
   - System performance (CPU, memory, latency)
   - Exchange connectivity status
   - Error rates and exception counts
   - Queue depths and processing rates

2. **Enhanced /api/status**: Add more detailed fields:
   - Last trade timestamp and details
   - Current position size and P&L
   - Risk metrics (VaR, drawdown)
   - Strategy-specific indicators

3. **New /api/runtime/logs Endpoint**: Recent application logs with filtering:
   - Error/warning logs from last N minutes
   - Trade execution logs
   - System event logs

4. **New /api/runtime/performance Endpoint**: Performance profiling data:
   - Function execution times
   - Database query performance
   - WebSocket message rates

## Implementation Details
- Use Pydantic models for structured responses
- Implement proper error handling and fallbacks
- Ensure all endpoints are public (no auth required) for dashboard access
- Add comprehensive logging for new metrics collection
- Use async/await for non-blocking metric collection
- Cache expensive metrics with TTL
- Include unit conversions (seconds to human-readable, bytes to MB, etc.)

## Testing Requirements
- Create unit tests for all new endpoints
- Test with mock bot manager and single bot scenarios
- Verify metrics accuracy against known test data
- Test error conditions and fallbacks
- Run all tests on Docker using 'make test'
- Ensure 100% test coverage for new code

## Acceptance Criteria
- All new endpoints return 200 with valid JSON
- Metrics are accurate and update in real-time
- No performance degradation (<5% overhead)
- Full backward compatibility with existing /api/status
- Comprehensive unit test suite passing on Docker
